# Deploy Button Spec

> One-click deploy produces a working Tasche instance. The user sets 3 secrets and signs in.

**Status:** Implemented (2026-03-01), simplified onboarding (2026-03-03)

---

## User Experience

Click the Deploy button → Cloudflare provisions everything → visit the URL → set 3 secrets in the dashboard → sign in.

No CLI, no terminal, no `wrangler` commands needed for deploy-button users.

### What Cloudflare auto-creates

The deploy button provisions all infrastructure automatically:

| Resource | Binding | Notes |
|----------|---------|-------|
| D1 database | `DB` | Database name chosen by Cloudflare |
| R2 bucket | `CONTENT` | Bucket name chosen by Cloudflare |
| KV namespace | `SESSIONS` | Namespace ID assigned |
| Queue | `ARTICLE_QUEUE` | Producer + consumer wired |
| Workers AI | `AI` | Just a binding declaration |
| Static Assets | `ASSETS` | Built from `./assets/` |

### What the app auto-detects at runtime

| Config | How | Where |
|--------|-----|-------|
| `SITE_URL` | Derived from request `Host` header when empty | `src/auth/routes.py:_get_site_url()` |
| `TTS_MODEL` | Defaults to `"melotts"` in code | `src/tts/processing.py` |

### What the user must provide (3 secrets)

These are the only manual steps, done in the Cloudflare dashboard under **Workers & Pages > Settings > Variables and Secrets**:

| Secret | Value | Why it can't be inferred |
|--------|-------|--------------------------|
| `GITHUB_CLIENT_ID` | From a GitHub OAuth App | User must create the OAuth App at [github.com/settings/developers](https://github.com/settings/developers) |
| `GITHUB_CLIENT_SECRET` | From the same OAuth App | Generated by GitHub |
| `ALLOWED_EMAILS` | User's GitHub email | Determines who can log in |

### Optional enhancements

| Secret | Purpose | Fallback without it |
|--------|---------|---------------------|
| `CF_ACCOUNT_ID` | Browser Rendering for JS-heavy pages | Plain HTTP fetch |
| `CF_API_TOKEN` | Browser Rendering API access | Plain HTTP fetch |
| READABILITY service binding | Mozilla Readability extraction | BeautifulSoup (~70% fidelity) |

## How the Deploy Button Works

When a user clicks the button, Cloudflare's deploy flow:

1. **Forks the repository** into the user's GitHub account
2. **Parses `wrangler.jsonc`** to determine required resources
3. **Auto-provisions resources** — D1, R2, KV, Queues, AI (no resource IDs needed)
4. **Shows setup page** — prompts for secrets listed in `.dev.vars.example`, with descriptions from `package.json` `cloudflare.bindings`
5. **Runs build** — `npm run build` from root `package.json` (builds the Preact frontend)
6. **Runs deploy** — `npm run deploy` from root `package.json` (applies D1 migrations, then deploys via pywrangler)
7. **Enables Workers Builds** — CI/CD auto-deploys on every push to main

### What it does NOT do

- Deploy Service Binding target Workers (e.g. readability-worker)
- Support `--env` flags (always targets the default environment)
- Handle multi-worker monorepos in a single click

## First-Boot Experience

After deploy, visiting the app shows a setup checklist (Login view calls `/api/health/config`). The checklist shows which secrets are configured and which are missing, with instructions for each.

| Item | Source | Required |
|------|--------|----------|
| GitHub OAuth App created | `GITHUB_CLIENT_ID` present | Yes |
| Client secret set | `GITHUB_CLIENT_SECRET` present | Yes |
| Email whitelist configured | `ALLOWED_EMAILS` non-empty | Yes |
| Readability Worker deployed | `READABILITY` binding present | No |
| Browser Rendering enabled | `CF_ACCOUNT_ID` + `CF_API_TOKEN` present | No |

Once all required items are green, the user reloads and sees the sign-in button.

## Post-Deploy Steps

### For deploy-button users (dashboard only)

1. **Create a GitHub OAuth App** at [github.com/settings/developers](https://github.com/settings/developers):
   - **Homepage URL:** your workers.dev URL (shown after deploy)
   - **Authorization callback URL:** `<your-url>/api/auth/callback`

2. **Add 3 secrets** in the Cloudflare dashboard (**Workers & Pages > your worker > Settings > Variables and Secrets**):
   - `GITHUB_CLIENT_ID` — from the OAuth App
   - `GITHUB_CLIENT_SECRET` — from the OAuth App
   - `ALLOWED_EMAILS` — your GitHub email

3. **Reload the page** — sign in with GitHub.

### Optional: Deploy readability-worker (CLI)

For better content extraction on complex pages. Requires cloning the forked repo:

```bash
cd readability-worker
npm install
npx wrangler deploy
```

Then add the service binding in the Cloudflare dashboard or `wrangler.jsonc`:

```jsonc
"services": [{ "binding": "READABILITY", "service": "readability-worker", "entrypoint": "ReadabilityService" }]
```

### Optional: Enable Browser Rendering (dashboard)

Add two more secrets in the dashboard:

- `CF_ACCOUNT_ID` — your Cloudflare account ID
- `CF_API_TOKEN` — an API token with Browser Rendering permissions

## Key Files

| File | Role in deploy flow |
|------|---------------------|
| `package.json` (root) | Build/deploy scripts + secret descriptions for the setup UI |
| `.dev.vars.example` | Lists secrets the deploy flow prompts for |
| `wrangler.jsonc` | Resource declarations (default env has no env-specific IDs) |
| `frontend/src/views/Login.jsx` | First-boot setup checklist via `/api/health/config` |
| `src/auth/routes.py` | `_get_site_url()` auto-detection from Host header |

### Root `package.json` structure

```json
{
  "scripts": {
    "build": "cd frontend && npm install && npm run build",
    "deploy": "npx wrangler d1 migrations apply DB --remote && npx pywrangler deploy"
  },
  "cloudflare": {
    "bindings": {
      "GITHUB_CLIENT_ID": { "description": "..." },
      "GITHUB_CLIENT_SECRET": { "description": "..." },
      "ALLOWED_EMAILS": { "description": "..." }
    }
  }
}
```

**Design decisions:**
- D1 migrations use the binding name `DB`, not the database name — binding names are stable across deployments while database names are user-chosen during provisioning.
- `--remote` flag is essential — applies migrations to the deployed D1 instance, not a local one.
- `cloudflare.bindings` descriptions support Markdown and are shown in the deploy setup UI.

## Workers Builds CI/CD

After the initial deploy, Workers Builds is automatically configured. Every push to main triggers:

1. `npm run build` — builds the Preact frontend into `./assets/`
2. `npm run deploy` — applies D1 migrations + deploys via pywrangler

## Design Decisions

### SITE_URL auto-detection

Users can't know their workers.dev URL before deploying, but `SITE_URL` drives OAuth callbacks and cookie security. Solution: `_get_site_url()` in `src/auth/routes.py` derives the URL from the request's `Host` header when the configured value is empty or contains the placeholder `<your-subdomain>`. This works for workers.dev and custom domains alike.

**Caveat:** The GitHub OAuth App callback URL must still match. Auto-detection ensures the app's own `redirect_uri` matches what it sends to GitHub, but the OAuth App config is manual.

### READABILITY service binding absent by default

The readability-worker must be deployed separately. Deploy-button users won't have it. The default `wrangler.jsonc` environment omits the `services` array — the binding is simply absent. `src/articles/extraction.py` falls back to BeautifulSoup automatically.

### Committed `assets/` directory

The built frontend is committed to `./assets/`. The deploy button's build step overwrites with a fresh build. Manual `pywrangler deploy` works without a prior build step. Stale assets in git are a minor cosmetic issue.

### Deploy button URL includes `&paid=true`

Tasche requires the Workers Paid plan ($5/month) — D1, R2, Queues, and Workers AI all require it. The `&paid=true` parameter sets user expectations before they start the deploy flow.
